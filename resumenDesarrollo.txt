ÚLTIMA IMPLEMENTACIÓN: Refactorización Cliente Multicomercio
================================================================================
Fecha: 2025-12-04
Branch: hardcore-wright

DESCRIPCIÓN:
------------
Se refactorizó completamente el modelo de datos para permitir que un usuario
(persona física con email único) pueda ser cliente de múltiples comercios,
manteniendo un solo login y múltiples vinculaciones independientes.

OBJETIVO LOGRADO:
-----------------
✓ Un usuario con email único puede ser cliente de múltiples comercios
✓ Un solo login (email + contraseña) para acceder a todos sus comercios
✓ Cada vinculación (Cliente) tiene su propio estado de aprobación
✓ Cada vinculación tiene su propia cuenta corriente
✓ Usuario.ComercioId = null para clientes (se relacionan vía tabla Cliente)
✓ Cliente actúa como tabla intermedia entre Usuario y Comercio

MODELO ANTES:
-------------
Usuario (ComercioId NOT NULL) 1:1 → Comercio
Usuario (email único) 1:N → Cliente (pero solo 1 activo por comercio)
Problema: No se podía crear otro Usuario con el mismo email

MODELO DESPUÉS:
---------------
Usuario (ComercioId nullable) 0..1 → Comercio (solo para Admin/UsuarioComercio)
Usuario (email único) 1:N → Cliente (múltiples activos) N:1 → Comercio

Cliente ahora es tabla intermedia:
Cliente {
  UsuarioId (required FK)
  ComercioId (required FK)
  EstadoId
  CuentaCorriente (1:1)
  Alias, NotasComercio (datos específicos de esta vinculación)
}

FLUJOS DE USUARIO IMPLEMENTADOS
================================================================================

FLUJO 1: Cliente se vincula a otro comercio
--------------------------------------------
1. Juan (usuario existente, cliente de Comercio A) hace login
2. Ve sus comercios: GET /api/clientes/mis-comercios
3. Selecciona "Vincularme a otro comercio"
4. POST /api/clientes/vincular { comercioId: 2, requiereAprobacion: true }
5. Sistema:
   - Valida que Juan no esté ya vinculado a Comercio B
   - Crea Cliente (UsuarioId=Juan, ComercioId=2, EstadoId=1 Pendiente)
6. Admin de Comercio B ve cliente pendiente
7. POST /api/clientes/{id}/aprobar
8. Sistema crea CuentaCorriente para Comercio B
9. Juan ahora puede operar en ambos comercios

FLUJO 2: Comercio agrega cliente existente
-------------------------------------------
1. Admin de Comercio B: "Agregar cliente"
2. Ingresa email: juan@example.com (ya es cliente de Comercio A)
3. POST /api/clientes { email: "juan@example.com", nombre: "Juan", ... }
4. Sistema:
   - Busca usuario por email → ENCUENTRA a Juan
   - Verifica que no esté ya en Comercio B
   - Crea vinculación Cliente (UsuarioId existente, ComercioId=2, EstadoId=2 Activo)
   - Crea CuentaCorriente para Comercio B
5. Juan puede operar inmediatamente en Comercio B
6. Próximo login de Juan: verá ambos comercios en la lista

FLUJO 3: Login con múltiples comercios
---------------------------------------
1. Juan hace login con juan@example.com
2. Sistema:
   - Busca usuario
   - Obtiene vinculaciones: Cliente(Comercio A, Activo), Cliente(Comercio B, Activo)
   - Genera token con claim "ComercioIds: 1,2"
3. Retorna LoginResponse:
   {
     "token": "...",
     "comercios": [
       { "id": 1, "nombre": "Ferretería López" },
       { "id": 2, "nombre": "Farmacia Pérez" }
     ]
   }
4. Frontend muestra selector de comercio
5. Juan selecciona y opera en el contexto elegido

FLUJO 4: Autogestión con usuario existente
-------------------------------------------
1. Juan (ya cliente de Comercio A) quiere registrarse en Comercio C
2. POST /api/auth/register/cliente
   {
     "email": "juan@example.com",
     "password": "suPassword",
     "comercioId": 3,
     ...
   }
3. Sistema:
   - Busca usuario por email → ENCUENTRA a Juan
   - Valida contraseña (debe ser la misma)
   - Verifica que no esté ya en Comercio C
   - Crea vinculación Cliente (EstadoId=1 Pendiente)
4. Retorna: "Pendiente de aprobación por el comercio"
5. Admin de Comercio C aprueba
6. Juan puede acceder a Comercio C

   - Datos específicos de cada vinculación
   - Comercio puede personalizar cómo ve al cliente
   - No afecta al usuario en otros comercios

DESCRIPCIÓN:
------------
Se implementó un sistema completo de registro que permite crear cuentas de dos
tipos diferentes: Comercios y Clientes. La solución utiliza DOS BOTONES
SEPARADOS en la interfaz de registro para proporcionar claridad inmediata al

DECISIÓN DE DISEÑO:
-------------------
✓ DOS BOTONES SEPARADOS (Implementado)
  - Botón 1: "Registrar Comercio" - Para negocios
  - Botón 2: "Registrar Cliente" - Para clientes de comercios existentes

  Ventajas:
  • Claridad inmediata sobre qué se va a crear
  • Menos pasos en el flujo de registro
  • Mejor UX para perfiles claramente diferenciados
  • No hay ambigüedad ni decisiones adicionales

✗ UN SOLO BOTÓN + SELECT (Rechazado)
  - Requiere pasos adicionales
  - Mayor fricción en el proceso

DESCRIPCIÓN:
------------
Se implementó un sistema completo de aprobación de clientes que diferencia entre
clientes creados por administradores (aprobados automáticamente) y clientes que
se auto-registran (requieren aprobación manual del comercio).

DECISIÓN ARQUITECTÓNICA CLAVE:
-------------------------------
✓ TABLA USUARIO UNIFICADA (Implementado)
  - Usuarios con rol "Cliente" autentican en tabla Usuario
  - Cliente.UsuarioId es FK nullable a Usuario
  - Beneficios:
    • Sistema de autenticación unificado
    • Mejor arquitectura a largo plazo
    • Facilita implementación de roles
    • No duplica lógica de passwords

✗ PASSWORDHASH EN CLIENTE (Rechazado)
  - Habría duplicado lógica de autenticación
  - Difícil de mantener

DECISIONES DE DISEÑO
================================================================================

1. **Tabla de Estados Extensible**
   ✓ Por qué: Permite agregar nuevos estados en el futuro sin cambiar código
   Ejemplos futuros: "Suspendido", "En Revisión", "Bloqueado"

2. **UsuarioId Nullable en Cliente**
   ✓ Por qué: Permite clientes creados por admin sin usuario inicialmente
   ✓ Clientes de autogestión siempre tienen UsuarioId
   ✓ Clientes de admin pueden tener UsuarioId si se les crea credenciales

3. **Cuenta Corriente en Aprobación**
   ✓ Por qué: No tiene sentido tener cuenta si no está aprobado
   ✓ Evita cuentas "fantasma" de clientes rechazados
   ✓ Admin-created clients: aprobación automática, cuenta inmediata

4. **OrigenRegistro como Int**
   ✓ Por qué: Más eficiente que string, extensible
   ✓ Futuro: agregar origen "Importación", "API", etc.

5. **AprobadoPorUsuarioId para Rechazos**
   ✓ Por qué: Mantener auditoría de quién rechazó
   ✓ Útil para reportes y seguimiento

6. **ON DELETE RESTRICT en FKs**
   ✓ Por qué: Previene eliminación accidental en cascada
   ✓ Seguridad: no se puede borrar estado si tiene clientes
   ✓ No se puede borrar usuario que aprobó clientes

7. **Página Separada para Pendientes**
   ✓ Por qué: Mejora UX, admin puede enfocarse en aprobaciones
   ✓ Alternativa: solo filtro en lista principal también disponible

IMPLEMENTACIÓN: Expiración de Sesión con Logout Automático
================================================================================
Fecha: 2026-03-26
Branch: 29-refresh-de-tocken-y-logout

DESCRIPCIÓN:
Se implementó el manejo correcto de la expiración de sesión JWT en la aplicación.
Cuando el token expira, el usuario es deslogueado automáticamente. No se realiza
refresh de token — la expiración siempre resulta en logout.

Se corrigió además un bug crítico donde el token expiraba en 24 segundos
en lugar de 24 horas por usar AddSeconds en lugar de AddHours.

ARCHIVOS CREADOS:
1. SaasACC.BlazorWasm/Services/AuthorizationMessageHandler.cs
   - DelegatingHandler que intercepta todas las respuestas HTTP del cliente
   - Si detecta un 401 Unauthorized, llama a IAuthService.Logout() automáticamente
   - Usa IServiceProvider para resolver IAuthService de forma diferida y evitar
     dependencia circular (HttpClient → Handler → IAuthService → HttpClient)

ARCHIVOS MODIFICADOS:
1. SaasACC.Application/Services/AuthService.cs
   - Bug fix: cambiado AddSeconds(jwtExpiryHours) → AddHours(jwtExpiryHours)
   - El token ahora expira correctamente en 24 horas según appsettings.json

2. SaasACC.BlazorWasm/Services/CustomAuthStateProvider.cs
   - Agregado método IsTokenExpired(): decodifica el payload Base64 del JWT,
     extrae el claim "exp" (Unix timestamp) y lo compara con DateTime.UtcNow
   - GetAuthenticationStateAsync() ahora verifica expiración antes de autenticar:
     si el token está vencido, limpia localStorage y devuelve estado anónimo

3. SaasACC.BlazorWasm/Program.cs
   - Reemplazado registro manual de HttpClient (new HttpClient{...}) por
     AddHttpClient("API") con encadenamiento de AuthorizationMessageHandler
   - El HttpClient resuelto desde IHttpClientFactory mantiene compatibilidad
     con el resto de los servicios sin necesidad de cambios adicionales

FLUJO IMPLEMENTADO:
1. App carga / recarga:
   CustomAuthStateProvider.GetAuthenticationStateAsync()
   → ¿Token en localStorage? NO → usuario anónimo
   → ¿Token expirado? SÍ → limpia localStorage → usuario anónimo
   → Token válido → autenticar normalmente

2. Request HTTP en runtime:
   AuthorizationMessageHandler.SendAsync()
   → request pasa al servidor
   → ¿Respuesta 401? SÍ → authService.Logout() → redirige al login
   → Otra respuesta → devuelve al componente normalmente

DECISIONES DE DISEÑO:
- No se implementó refresh de token: la expiración siempre resulta en logout.
  Razón: simplicidad y seguridad (no hay tokens de refresh que gestionar).
- IServiceProvider en el handler en lugar de inyección directa de IAuthService:
  evita dependencia circular en el contenedor de DI de Blazor WASM.
- IsTokenExpired() es puramente local: no hace requests al servidor, decodifica
  directamente el payload Base64 del JWT (reutiliza ParseBase64WithoutPadding).

BUGS CORREGIDOS:
- Token expiraba en 24 segundos en lugar de 24 horas (AddSeconds → AddHours)
